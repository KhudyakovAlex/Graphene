# AI Development Guide

Этот документ читает AI, который разрабатывает или изменяет саму Graphene.
Он является основным руководством для AI-агентов в этом репозитории.

## Назначение

Graphene - дизайн-система для Vue-приложений организации.
Цель проекта - единый стиль интерфейсов, переиспользуемые Vue-компоненты, дизайн-токены и правила верстки.

Дизайн-источник - Pixso. Кодовая реализация - Graphene.

## Среда Эксплуатации

Приложения на Graphene должны работать без доступа к внешнему интернету.
Допустимы только локальные ресурсы приложения и ресурсы внутреннего интранета организации.

Нельзя полагаться на CDN, внешние шрифты, внешние иконки, внешние изображения, публичные API или загрузку пакетов во время работы приложения.

## Основной Принцип

Не рисовать интерфейс заново в каждом месте.
Нужно собирать интерфейсы из токенов, базовых компонентов и составных компонентов.

Визуальный характер Graphene ближе к строгим промышленным интерфейсам в духе IBM Carbon: сетка, ясные плоскости, минимум декоративности.
Для большинства элементов базовое скругление - `--g-radius-none`.
Скругления используются только как осознанное решение из Pixso или правила компонента.

## Темы

Graphene должна поддерживать светлую и темную темы.

Тема решается через семантические токены, а не через хардкод цветов в компонентах.

`src/tokens/` хранит:

- base tokens: базовую палитру и другие сырые значения;
- семантические токены: смысловые переменные интерфейса, например фон страницы, основной текст, граница, фон опасного действия;
- значения семантических токенов для светлой и темной темы.

`src/styles/` применяет семантические токены к `body`, базовым HTML-элементам и глобальным состояниям.
`src/styles/` не должен заново определять палитру.

Компоненты должны использовать семантические токены, а не базовые цвета.
Например, использовать `--g-text-primary`, а не `--g-gray-950`.

Семантические токены и их привязка к базовым токенам задаются только в Pixso.
AI не должен вручную выбирать или менять в коде, к какому base token привязан semantic token.
AI не должен добавлять новый semantic token в код и вручную привязывать его к base token.
Если для новой роли компонента нужен semantic token, сначала нужно остановиться и попросить владельца добавить этот token в Pixso.
Если компонент визуально не совпадает с Pixso при текущих токенах, AI должен сказать владельцу, какой semantic token и как нужно поправить в Pixso.
После правки в Pixso AI только синхронизирует токены из Pixso в код.
Компонент в этом случае меняется только структурой, состояниями и использованием уже существующих semantic tokens.

Переключение темы должно работать через атрибут:

`data-theme="light"`

или:

`data-theme="dark"`

`examples` должен уметь переключать светлую и темную темы и показывать компоненты в обоих режимах.

В Pixso режимы переменных должны называться строго:

- `light`
- `dark`

Один Pixso token с режимами `light` и `dark` превращается в одну CSS-переменную с разными значениями:

`light` -> `[data-theme='light']`

`dark` -> `[data-theme='dark']`

## Устройства И Адаптивность

Приложения на Graphene должны работать на разных устройствах: десктопах, планшетах и телефонах.

Нужно учитывать полноэкранную оболочку приложения, похожую на Electron, включая мобильные устройства.

Интерфейсы должны поддерживать портретную ориентацию телефона.

Компоненты и раскладки не должны предполагать только широкий десктопный экран.
Минимум нужно продумывать:

- перенос и сжатие панелей;
- удобные размеры интерактивных элементов для touch;
- отсутствие горизонтального скролла там, где он не является осознанным поведением;
- корректную работу модальных окон и меню на узких экранах;
- читаемость текста и доступность основных действий на телефоне.

`examples` должен показывать, как компоненты и типовые экраны ведут себя на узких экранах.

## Структура

`src/tokens/`

Дизайн-токены: цвета, размеры, отступы, типографика, радиусы, тени, z-index и другие базовые значения.

`src/tokens/base/`

Базовые токены: сырые значения дизайна без привязки к конкретному месту интерфейса.
Сюда входят цвета, радиусы, отступы, размеры, тени и другие фундаментальные значения.

`src/tokens/semantic/`

Семантические токены: смысловые роли интерфейса. Компоненты должны использовать их вместо базовых цветов напрямую.
Для состояний кнопок используются `--g-action-*`.
Для полей ввода используются `--g-field-*`.
Для чекбоксов используются `--g-checkbox-*`.
Для радиокнопок используются `--g-radio-*`.

Семантические токены могут быть разделены по типу значения:

- `colors.css` - цветовые роли;
- `typography.css` - типографические роли;
- `size.css` - размерные роли.

Все semantic tokens и их привязка к base tokens должны приходить из Pixso.

`src/styles/`

Глобальные стили: reset, базовая типографика, темы, подключение токенов.
`src/styles/base.css` - основная точка подключения базового внешнего вида Graphene.

`src/assets/fonts/`

Локальные файлы шрифта IBM Plex Sans. Graphene не должна загружать шрифты из внешнего интернета.

`src/assets/icons/`

Отобранный локальный набор иконок Graphene. Он пополняется по мере развития дизайн-системы.
Иконки должны храниться локально и не должны загружаться из внешнего интернета.
Файлы иконок должны называться в `kebab-case`, например `search.svg`.
SVG-иконки должны использовать `currentColor`, чтобы цвет задавался через CSS и semantic tokens, а не хардкодился внутри SVG.

`src/assets/icons/others/`

Большой фиксированный запасной набор иконок на случай редких сценариев.
Иконки из `others/` не считаются основным отобранным набором Graphene.
Если иконка становится регулярной частью дизайн-системы, ее нужно перенести в `src/assets/icons/`.
При выборе иконки сначала ищи в `src/assets/icons/`; используй `others/` только если подходящей иконки нет в основном наборе.

`src/components/base/`

Простые базовые Vue-компоненты. Они должны быть максимально универсальными и предсказуемыми.
Сейчас есть `BaseButton`, `BaseInput`, `BaseTextarea`, `BaseSelect`, `BaseSlider`, `BaseCheckbox` и `BaseRadio`.
`BaseInput` поддерживает правую зону действия через slot `trailing`.
Иконки для `BaseInput` нужно передавать через `trailing`, например `search.svg` из `src/assets/icons/`.
`BaseInput` не должен показывать нативный browser clear-крестик у `type="search"`; правая зона управляется Graphene через `trailing`.
`BaseTextarea` - многострочный вариант текстового ввода на тех же `--g-field-*` токенах, без отдельного визуального стиля.
`BaseSelect` - нативный select на тех же `--g-field-*` токенах и размерах `sm`, `md`, `lg`; поддерживает `loading`, `clearable` и `multiple`.
`BaseSlider` - одинарный slider на `--g-slider-*` токенах, без встроенного label и helper.
`BaseCheckbox` не содержит встроенный label, потому что текущий Pixso-дизайн чекбокса описывает только сам контрол.
`BaseRadio` не содержит встроенный label, потому что текущий Pixso-дизайн радиокнопки описывает только сам контрол.

`src/components/layout/`

Компоненты структуры экрана: контейнеры, панели, области, раскладки.

`src/components/data/`

Компоненты отображения данных: списки, таблицы, деревья, свойства, пагинация.

`src/components/feedback/`

Компоненты обратной связи: уведомления, ошибки, загрузка, пустые состояния, модальные окна.

`src/components/navigation/`

Компоненты навигации: вкладки, меню, хлебные крошки, панели действий.

`docs/`

Документация для дизайн-системы: правила, рекомендации и описание компонентов. Не является частью исходного кода библиотеки.

`docs/human/`

Документация для людей.

`docs/ai/`

Документация для AI-агентов.

`docs/ai/usage-guide.md`

Спецификация для AI, который использует Graphene при создании интерфейсов приложений.

`docs/ai/development-guide.md`

Спецификация для AI, который разрабатывает или изменяет саму Graphene.

`examples/`

Локальная Vue/Vite-витрина Graphene: примеры компонентов и типовых экранов. Не должна содержать копии токенов, стилей или компонентов из `src/`.
Компоненты витрины должны импортироваться из `src/components/`.
Корневой `examples/index.html` - основной локальный сайт витрины. Он должен открываться напрямую из файловой системы в браузере без dev-сервера.
`npm run dev` в `examples` нужен только для разработки, а не для использования готовой витрины.
После изменения Vue-кода витрины нужно обновлять локальный bundle через `cd examples && npm run build`.
Если `examples/dist` не пересобран, `examples/index.html` покажет старую версию, даже если исходники в `examples/src/` уже изменены.

`examples/src/`

Vue-код витрины. Он показывает, как использовать элементы Graphene, но не является частью библиотеки компонентов.

`examples/src/example-data/`

Демонстрационные данные для витрины.

## Правила Разработки

1. Использовать Vue 3.
2. Компоненты писать как `.vue` Single File Components.
3. Использовать TypeScript.
4. В `.vue` компонентах использовать `<script setup lang="ts">`.
5. Типизировать `props`, `emits` и публичные типы компонентов.
6. Предпочитать Composition API и `<script setup>`.
7. Стили компонента держать рядом с компонентом в `<style scoped>`, если стиль не является глобальным.
8. Глобальные значения выносить в токены и `src/styles/`.
9. Не задавать цвета, отступы, размеры и радиусы произвольно, если есть подходящий токен.
10. Не использовать inline styles без сильной причины.
11. Не создавать локальный аналог существующего компонента.
12. Не смешивать дизайн-систему с бизнес-логикой конкретного приложения.
13. Компоненты должны быть предсказуемыми по `props`, `emits` и `slots`.
14. Все ассеты, шрифты, иконки и runtime-зависимости должны поставляться локально или через утвержденный внутренний контур.
15. Основной шрифт Graphene - IBM Plex Sans. Подключение шрифта держать в `src/styles/fonts.css`, файлы шрифта - в `src/assets/fonts/`.
16. Не оставлять нативные browser controls, которые визуально выбиваются из Graphene. Например, у `input[type="search"]` нужно отключать нативный clear-крестик и управлять правой зоной через компонентный API или slot.

## Работа С Pixso

1. Считать Pixso источником визуального решения.
2. Получать данные из Pixso через MCP.
3. Сразу раскладывать результат по папкам проекта: токены в `src/tokens/`, стили в `src/styles/`, компоненты в `src/components/`, документацию в `docs/`.
4. Переносить повторяемые значения из Pixso в design tokens.
5. Если значение встречается многократно, не хардкодить его внутри компонента.
6. Если макет противоречит существующим токенам или компонентам, зафиксировать вопрос в `docs/`.
7. Не добавлять новый визуальный стиль без явного основания из Pixso или решения владельца системы.

## Добавление Нового Компонента

1. Сначала получить выбранный узел из Pixso через MCP.
2. Посмотреть screenshot и generated code только как источник фактов о структуре, состояниях, размерах и токенах.
3. Проверить, существуют ли нужные semantic tokens в Pixso.
4. Если semantic tokens не хватает, остановиться до написания компонента. Не добавлять semantic tokens в код вручную и не выбирать для них base tokens. Нужно сказать владельцу, какие semantic tokens добавить или изменить в Pixso.
5. После правки Pixso выкачать токены и синхронизировать их в `src/tokens/semantic/`.
6. Только после этого писать или менять Vue-компонент.
7. Компонент должен использовать semantic tokens своей роли, например `--g-field-*` или `--g-checkbox-*`, а не base tokens вроде `--g-gray-*`.
8. Не добавлять label, helper, icon, group behavior или другие части API, если их нет в Pixso-дизайне или явном решении владельца.
9. Компонент разместить в правильной категории, например `src/components/base/BaseCheckbox.vue`.
10. Экспортировать компонент из `src/components/base/index.ts`.
11. Добавить пример в `examples/src/views/BaseComponentsView.vue` или другую подходящую страницу витрины.
12. Обновить документацию компонента и AI-документацию, если изменились правила или доступные компоненты.
13. После изменения Vue-кода витрины выполнить `cd examples && npm run build`, чтобы локальный `examples/index.html` открывался без dev-сервера.

Перед написанием компонента проверь:

- semantic tokens для всех визуальных ролей компонента уже есть в Pixso;
- размеры и состояния компонента понятны из Pixso;
- disabled-состояние соответствует существующему подходу Graphene, например через `--g-opacity-disabled`, если Pixso не задает отдельную semantic role;
- компонент не оставляет нативные browser controls, которые нельзя стилизовать токенами Graphene;
- пример в `examples` после изменения будет пересобран в `examples/dist`.

## Работа С Токенами

1. Base tokens - сырые значения. Их можно использовать только внутри token files или когда semantic token для роли еще не определен и владелец явно разрешил временное решение.
2. Semantic tokens - единственный нормальный источник цветов, размеров и типографических ролей для компонентов.
3. AI не должен вручную менять привязку semantic token к base token в коде.
4. Если в Pixso semantic token имеет неверное значение, AI должен назвать владельцу token и ожидаемую правку в Pixso.
5. После правки Pixso AI синхронизирует токены в код.
6. Если Pixso отдает очевидно ошибочное значение, например размер `0` для видимого компонента, AI должен остановиться и сообщить об этом владельцу, а не молча придумывать значение.

## Правила Для AI

1. Сначала ищи существующий компонент или токен.
2. Если подходящий компонент есть, используй или расширяй его, а не создавай новый.
3. Если создаешь компонент, размещай его в правильной категории.
4. Для нового компонента добавляй минимальную документацию или пример в `examples`, если это уместно.
5. Не придумывай случайные CSS-значения.
6. Не копируй стили между компонентами, если их можно выразить токенами.
7. Не делай крупные рефакторинги без необходимости.
8. Сохраняй простые имена компонентов: `BaseButton`, `BaseInput`, `AppShell`, `DataTable`, `EmptyState`.
9. Пиши код так, чтобы его могли использовать разные команды, а не один конкретный проект.
10. Не добавляй зависимости от внешнего интернета.
11. При изменении правил реализации обновляй документацию в `docs/`.
12. Не дублируй токены, стили и компоненты внутри `examples`; витрина должна импортировать их из `src/`.
13. Если меняешь правила, структуру, темы, токены, компоненты или подход к examples, обновляй `docs/ai/usage-guide.md` и `docs/ai/development-guide.md`.

## Ожидаемый Результат

Graphene должна давать разработчикам:

- единые дизайн-токены;
- готовые Vue-компоненты;
- правила верстки;
- демо-страницы;
- документацию;
- понятные ограничения для AI-генерации интерфейсов.
